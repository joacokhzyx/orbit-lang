# Contributing to Orbit

Thank you for your interest in contributing to Orbit! We welcome contributions from developers of all skill levels. This document outlines the process for proposing changes, submitting pull requests, and maintaining code quality standards across the Orbit compiler and C runtime engine.

---

## Code of Conduct

All contributors are expected to adhere to the [Orbit Code of Conduct](CODE_OF_CONDUCT.md). Please treat all members of the community with respect and professionalism.

---

## Development Environment Setup

### Prerequisites

To build and test Orbit locally, ensure you have the following installed:

1. **Zig Compiler**: Version `0.13.0` or higher (available via system package managers or [ziglang.org](https://ziglang.org/download/)).
2. **C Compiler**: MSVC (Windows), GCC, Clang, or `zig cc`.
3. **Python 3.10+** (Optional, required for running the benchmark suite).

### Building from Source

Clone the repository and compile the Orbit binary:

```bash
git clone https://github.com/joacokhzyx/orbit-lang.git
cd orbit
zig build -Doptimize=ReleaseFast
```

The resulting `orbit` executable will be placed in `./zig-out/bin/orbit`.

## Running Tests & Benchmarks

### Unit & Integration Tests

Orbit maintains a comprehensive test suite in `src/tests.zig` covering lexing, parsing, semantic analysis, IR generation, C code generation, and runtime memory recycling.

Run the test suite:

```bash
zig test src/tests.zig
```

Ensure all tests pass before submitting a pull request.

### Running Marketing & Stress Benchmarks

To execute the multi-language stress and performance benchmark suite:

```bash
python benchmarks/marketing_suite/run_marketing_bench.py
```

## Coding Standards & Guidelines

Orbit follows strict architectural and diagnostic conventions modeled after `ziglang/zig` and `rust-lang/rust`.

### 1. Diagnostic Formatting & Error Messages

Diagnostic outputs generated by the compiler are critical for developer experience. Follow these strict rules when adding or modifying error messages:

- **No Borders or Emojis**: Error diagnostics must be clean text without decorative ASCII/Unicode box borders or emojis.
- **English Suggestions**: All compiler suggestions, hints, and error explanations **must be written in English**.
- **Verbose Explanations**: Provide descriptive, actionable error context rather than cryptic short messages.
- **Consistent Color Styling**: Use standard ANSI color codes (red for errors, yellow for warnings, cyan for line numbers).

### 2. Code Formatting

- **Zig Code**: All `.zig` files must be formatted using `zig fmt`. Run `zig fmt .` prior to committing.
- **C Runtime Engine**: Keep C code in `src/runtime/` idiomatic, const-correct, and free of compiler warnings (`-Wall -Wextra`).
- **Documentation Comments**: Use triple-slash doc comments (`///`) for public functions, structs, and module APIs.

## Pull Request Process

1. **Create a Feature Branch**:
   ```bash
   git checkout -b feature/your-feature-name
   ```
2. **Commit Changes**: Write clear, descriptive commit messages.
3. **Run Verification Suite**:
   ```bash
   zig fmt .
   zig test src/tests.zig
   ```
4. **Submit PR**: Open a pull request against the `main` branch with a clear description of the problem solved or feature added.
